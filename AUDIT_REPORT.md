## 📧 SMTP / Email

Из кода (`apps/api/src/env.ts`):
```
EMAIL_PROVIDER: z.enum(['mock', 'smtp']).default('mock')
```

**Дефолт — `mock`**. При `mock` коды верификации только **логируются** в консоль, письма не отправляются.

Если в production env переменных нет `EMAIL_PROVIDER=smtp` + `SMTP_USER` + `SMTP_PASSWORD` → **email-верификация будет тихо не работать** даже после деплоя нового auth.

#### 1. Регистрация пользователей полностью сломана
- **Симптом:** `/ru/sign-up` → 404. Зарегистрироваться невозможно.
- **Причина:** Коммит `798125e feat(aivita): full auth rebuild` **не задеплоен** в контейнер `aivita`.
- **Что починить:** Пересобрать и задеплоить контейнер `aivita` с последним кодом.

#### 2. Sign-in показывает старый UI (без пароля)
- **Симптом:** Страница входа показывает `"Войди через привычный сервис — без паролей"` + кнопки Google/Telegram вместо email+password формы.
- **Причина:** Тот же — старый контейнер.
- **Доп. эффект:** Даже если пользователь каким-то образом зарегистрировался — войти через новую форму невозможно (страница старая).

#### 3. API aivita-роуты не задеплоены
- **Симптом:** `POST /v1/aivita/auth/register` → 404, `GET /v1/landing/content/ru` → 404.

## 🎯 Топ-5 рекомендаций по приоритету

### 1. 🚨 Задеплоить все три контейнера заново (aivita + api + admin)

Ни один из трёх контейнеров не имеет последнего кода. Нужен **trigger rebuild всех трёх** через Coolify. Порядок:
1. `api` — сначала, т.к. фронтенд зависит от него
2. `aivita` — новый auth требует новых API роутов
3. `admin` — получит monitoring + CMS

### 2. ✉️ Проверить EMAIL_PROVIDER в prod

До или сразу после деплоя убедиться:
```
docker exec <api-container> env | grep -E "EMAIL_PROVIDER|SMTP_USER|SMTP_HOST"
```
Если `EMAIL_PROVIDER=mock` → пользователи не получат код верификации. Регистрация будет "зависать".

### 3. 🌱 Сделать seed для landing_content

Таблица `landing_content` (скорее всего) пустая. CMS в admin будет показывать пустые поля.  
Нужно написать seed-скрипт который заполняет её начальными данными из переводов или вручную.  
Пока landing_content пустой — публичный API `/v1/landing/content/ru` вернёт `{}` вместо текстов.

### 4. 🔗 Подключить лендинг к CMS API

Сейчас `MarketingPage` читает тексты только из статических переводов next-intl.  
CMS работает "вхолостую". Нужно добавить опциональный fetch из API в `MarketingPage` с fallback на статику.

### 5. 🧪 Тест полного flow после деплоя

После задеплоя новых контейнеров — вручную:
1. Зарегистрироваться (email + никнейм + пароль)
2. Получить код → ввести → попасть на /home
3. Проверить Health Score (реальный из API или fallback)
4. Зайти в /chat — отправить сообщение
5. Зайти в admin с farhodni@gmail.com — проверить dashboard, patients, monitoring, cms
